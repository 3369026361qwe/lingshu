 ### 2.3 消除 sys.path.insert 依赖

 **现状**: `run_daily_pipeline.py`、`run_full_period_backtest.py` 等脚本开头均包含：
 ```python
 sys.path.insert(0, str(Path(__file__).resolve().parent.parent))
 ```
 这与项目自身的导入约定（"禁止使用 sys.path.insert hack"）相矛盾。

 **修复方案**:
 1. 项目已有 `pyproject.toml`，其中 `[project]` 定义了 `name = "lingshu"`，但缺少 `[build-system]` 配置
 2. 添加 `[build-system]` 并执行 `pip install -e .`，使 `lingshu` 成为可导入的包
 3. 所有脚本改为 `from lingshu.xxx import ...` 的形式，彻底移除 `sys.path.insert`

 ```toml
 # pyproject.toml 补充
 [build-system]
 requires = ["setuptools>=64.0", "wheel"]
 build-backend = "setuptools.backends._legacy:_Backend"

 [tool.setuptools.packages.find]
 where = ["."]
 ```

 **涉及文件**: `scripts/*.py` 中约 10 个文件使用了 `sys.path.insert`

 ### 3.2 WebSocket 推送真实数据

 **现状**: 三个 WebSocket 端点（`/ws/market` / `/ws/agents` / `/ws/risk`）收到任何消息后均返回固定 Mock 数据：
 ```python
 # server.py L130
 await websocket.send_json({"type": "market", "data": {"index": "沪深300", "change": "+0.5%"}, ...})
 ```

 **改进方案**:
 1. 后台启动一个定时任务（`asyncio.Task`），每隔 N 秒从 `shuju/` 或 `shujuku/` 拉取最新市场数据
 2. Agent WebSocket 改为监听 `Orchestrator` 的分析完成事件（可用 `asyncio.Event` + 广播）
 3. 前端 WebSocket 客户端按 signal 类型处理，而非目前的通用消费模式

 ```python
 async def broadcast_market():
     while True:
         data = await repo.get_latest_market_snapshot()
         for ws in _ws_clients:
             await ws.send_json({"type": "market", "data": data})
         await asyncio.sleep(5)
 ```

 **涉及文件**: `jiekou/server.py`, `jiekou/middleware.py`

 ### 3.4 GNN 运行时实装

 **现状**: GNN 模型定义（`tushenjing/gnn_model.py`）清晰，但实际运行时依赖 NumPy 降级路径（`NumPyGCN` / `NumPyGAT`），因为 PyTorch + PyG 在 Windows 上安装困难。没有预训练权重文件，`graph_inference.py` 的推理结果实质来自简单矩阵运算而非真正的 GNN。

 **建议方案**:
 1. **短期**: 完善 `gnn_numpy.py` 的 NumPy 实现，确保其行为与 PyG 版本尽可能一致（权重初始化、前向传播的数值路径）
 2. **中期**: 在 Docker 镜像（Linux 容器）中安装 PyTorch + PyG，Linux 环境下安装问题少得多
 3. **长期**: 提供一组预训练权重文件（`*.pt`），`graph_inference.py` 默认加载

 ```dockerfile
 # Dockerfile.api 补充
 RUN pip install torch --index-url https://download.pytorch.org/whl/cpu
 RUN pip install torch-geometric
 ```

 **涉及文件**: `tushenjing/gnn_model.py`, `tushenjing/gnn_numpy.py`, `tushenjing/graph_inference.py`, `bushu/Dockerfile.api`

 ### 3.5 安装依赖验证 + CI/CD

 **现状**: `pyproject.toml` 列出了 20+ 依赖包，但在当前环境（Python 3.10 + Windows）安装会遇到问题：
 - `requires-python = ">=3.11"` → 环境是 3.10
 - `TA-Lib>=0.5.0` → Windows 需预编译 wheel（常见失败点）
 - `torch>=2.5.0` / `torch-geometric>=2.6.0` → GPU + Windows 组合安装复杂

 **改进步骤**:
 1. 创建 `requirements-ci.txt`，只包含轻量测试依赖（pytest, numpy, pandas, fastapi, sqlalchemy）
 2. 添加 GitHub Actions CI 配置，至少在 Linux + Python 3.11 上通过测试集合
 3. 分离可选依赖：`pip install -e ".[dev]"` 本地开发，CI 只安装核心 + 测试

 ```yaml
 # .github/workflows/ci.yml
 name: CI
 on: [push, pull_request]
 jobs:
   test:
     runs-on: ubuntu-latest
     steps:
       - uses: actions/checkout@v4
       - uses: actions/setup-python@v5
         with: { python-version: "3.11" }
       - run: pip install -e ".[dev]"
       - run: python -m pytest tests/ -q --tb=short
 ```

 **涉及文件**: 新建 `.github/workflows/ci.yml`, 新建 `bushu/requirements-ci.txt`

 ## 四、P2 — 按需优化

 ### 4.1 factor_fusion.py 精简因子集

 **现状**: `juece/factor_fusion.py` 的 `DEFAULT_WEIGHTS` 定义了 35 个因子，其中 `money_flow` / `downside_vol` / `var_95` / `vstd_20` / `vma_5` / `rank_20` 的权重仅为 0.005，在融合计算中几乎没有影响（合计贡献 0.03，占 35 因子的 3%）。

 **建议**: 将贡献极低的因子（`weight < 0.01`）移入 `DEFAULT_WEIGHTS_LOW_PRIORITY` 或加注释标记，降低认知负担。改为加权聚类策略（例如先按类别等权汇总，再类别间加权）。

 **涉及文件**: `juece/factor_fusion.py`

 ---

 ### 4.2 动量方向的实证说明

 **现状**: 所有动量因子方向均为 `-1`（反转策略），即认为过去涨 → 未来会跌。这在量化研究中是一个**有争议**的假设——A 股短周期动量效应存在的实证证据不少。

 **建议**: 在 `factor_fusion.py` 或因子配置文档中添加注释说明：
 ```python
 # 当前采用反转策略（direction=-1），依据为 A 股 2018-2025 年 Rank IC 检验结果
 # 如需切换为趋势跟踪，将 direction 改为 +1 并重新运行 factor_validation.py 验证
 ```

 **涉及文件**: `juece/factor_fusion.py`, `yinzi/momentum_factors.py`

 ---

 ### 4.3 Docker 健康检查 + 生产配置

 **现状**: `bushu/docker-compose.yml` 中缺少 healthcheck 配置，API 服务没有启动探测，容器编排在 `docker-compose up` 后可能因服务顺序问题失败。

 **建议**:

 ```yaml
 healthcheck:
   test: ["CMD", "curl", "-f", "http://localhost:8000/api/health"]

   timeout: 10s
   retries: 3
 ```

 **涉及文件**: `bushu/docker-compose.yml`

 ## 五、P3 — 代码风格与微量改进

 ### 5.1 前端 CSS 模块化

 **现状**: 5 个页面的组件大量使用内联 `style={{}}` 对象。`theme.css` 中定义的 CSS 变量利用率偏低。

 **建议**:
 1. 为每个页面创建 CSS Module（`*.module.css`）
 2. 将重复的内联样式抽取为类名
 3. 统一使用 `theme.css` 的 CSS 变量

 **涉及文件**: `qianduan/src/pages/*.tsx`, `qianduan/src/components/*.tsx`

 ---

 ### 5.2 lambda_rank.py 归属

 **现状**: `tushenjing/lambda_rank.py` 实现的 LambdaRank 损失函数是排序学习（LTR）模块，逻辑上属于 `juece/`（决策层）而非 `tushenjing/`（图神经引擎）。

 **建议**: 移动至 `juece/lambda_rank.py`，或在 `tushenjing/lambda_rank.py` 加跨模块引用转发。

 **涉及文件**: `tushenjing/lambda_rank.py` → `juece/lambda_rank.py`

 ---

 ### 5.3 factor_fusion.py 数据库依赖解耦

 **现状**: `juece/factor_fusion.py` 直接导入 `from shujuku.session import SessionContext` 和 `from sqlalchemy import text`，引入数据库耦合。`FactorFusion` 作为纯计算类不应感知数据库存在。

 **建议**: 因子权重从外部注入（通过构造函数参数传入），而非在 `FactorFusion` 内部查询数据库。

 ```python
 # 当前
 def load_weights_from_db(self) -> None:
     # 内部查询 SQL

 # 建议
 class FactorFusion:
     def __init__(self, weights: Optional[dict] = None):
         self._weights = weights or self.DEFAULT_WEIGHTS
 ```

 **涉及文件**: `juece/factor_fusion.py`
